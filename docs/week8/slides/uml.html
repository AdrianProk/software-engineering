Weitere Spezifikationsmöglichkeiten mit der UML

Marcel Lüthi
Departement Mathematik und Informatik

Herzliche Willkommen zu diesem Screencast zum Thema Spezifikation. Jedes nichttriviale System muss spezifiziert werden. Bevor wir eine Brücke bauen, müssen wir zuerst spezifizieren, welche Anforderungen denn die Brücke erfüllen muss. Soll sie nur für Fussgänger oder auch für Lastwagen ausgelegt sein? Soll die Brücke für die Ewigkeit gebaut werden, oder reicht es, wenn Sie 2 Jahre lang ihren Dienst tut? Muss sie einem Tsunami standhalten? Wir können aber nicht nur die Anorderung an die Brücke selbst, sondern auch für die Teile spezifizieren. Wie müssen die Brückenpfeiler beschaffen sein? Wie reissfest sollen die Seile und Kabel sein? Wir haben also nicht nur Spezifikationen für das gesamtsystem, sondern auch für die einzelnen Komponenten. Spezifikationen werden auf verschiedenen Ebenen eingesetzt.

Use-cases

Textuelle Beschreibung einer Systeminteraktion eines Akteur um fachliches Ziel zu erreichen.
  • Was muss gemacht werden um Ziel des Nutzers zu verwirklichen?
    • Einnahme von Nutzersicht hilft Anforderungen zu finden
  • Use-cases sollen nicht zu komplex sein
    • Kaffeepausentest: Anwendungsfall ist zu komplex, wenn Nutzer während Interaktion Kaffeepause braucht

Use-cases: Beispiel

Name: Buch ausleihen
Akteure: Bibliothekar, Kunde
Standardablauf:
  1. Kunde gibt Bibliothekar seine Karte
  2. Bibliothekar prüft Karte und schaut sich getätige Vorbestellung an
  3. Bibliothekar holt alle bestellten Bücher
  4. Bibliothekar vermerkt im System das Bücher vom Kunden ausgeliehen sind.

Endliche Automaten (Finite State Machines)

  • Eine Menge von Zuständen $Q$
  • Eine endliche Menge von inputs $I$
  • Transitionsfunktion $\phi : Q \times I \to Q$
    • $\phi$ kann partielle Funktion sein

UML Zustandsdiagramme

  • Vielfältige Modellierungsmöglichkeiten
  • Unterstützt verschachtelte Zustände und viele weitere Erweiterungen

UML Aktivitätsdiagramme

Umsetzung in UML: OCL

  • OCL: Object Constraint Language ( www.omg.org)
  • Designed um Eigenschaften/Einschränkungen in OO Systemen zu spezifizieren

Beispiel OO Design:

  • Beispiel: Wikipedia (https://de.wikipedia.org/wiki/Object_Constraint_Language)

    • Umsetzung in UML: OCL

    Beispiele von Constraints in OCL:
    • context Person inv: self.alter ≥ 0
    • context Person inv: self.eltern $\rightarrow$ forAll(e|e.alter>self.alter)
    • context Person inv: self.eltern->size() ≤ 2
    • context Person::bekommtKind()
      post: self.kinder->notEmpty()
          and self.kinder->size() > self.kinder@pre->size()
    • context Person inv: self.alter ≤ 18 implies self.autos->isEmpty()
    • context Auto inv: self.erstzulassung ≥ self.baujahr
    • context Person inv: Person.allInstances()->exists(p | p.autos->size() > 0)

    UML für Spezifikation

    Use-case Diagramm Start vom Projekt, zum Finden von top-level Funktionen
    Klassendiagramm Anfangen mit wichtigsten Entitäten. Detailieren während ganzem Projekt
    Sequenzdiagramm Dokumentieren wichtige Szenarien die mehrere Objekte betreffen. Fokus auf zeitlicher Abfolge.
    Kommunikationsdiagramm Wie Sequenzdiagramm aber Fokus auf Beziehung der Klassen.
    Zustandsdiagramm Spezifiziert Verhalten von Instanzen einer Klasse. Erst spät im Projekt einsetzen.
    OCL Kann zusammen mit jedem UML Modell verwendet werden um Einschränkungen zu spezifizieren.
